## 背景与架构

SORA-HUB 的请求链路：

```
用户浏览器 → 又拍云 CDN 节点 → Nginx → PHP-FPM → MySQL
```

项目中有两类资源，缓存策略完全不同：

| 资源类型 | 示例 | 缓存策略 |
|---|---|---|
| 静态资源 | CSS、JS、图片、字体 | 长期缓存（7 天），CDN 加速效果显著 |
| PHP API | `version-proxy.php`、`barrage.php` | **不缓存**，PHP 自身有数据库缓存逻辑 |

## 缓存配置

### 参数跟随 — 必须设为「全程跟随」

这是**最关键**的配置，直接决定项目能否正常运行。

#### 什么是参数跟随

CDN 默认会忽略 URL 中 `?` 之后的参数。例如：

```
用户请求: /api/version-proxy.php?tool=scripthookv
CDN 缓存键: /api/version-proxy.php（丢弃了 ?tool=scripthookv）
CDN 回源: /api/version-proxy.php（也丢弃了参数）
```

#### 为什么必须全程跟随

SORA-HUB 的 PHP API 完全依赖查询参数区分行为：

```
version-proxy.php?tool=scripthookv       → 返回 ScriptHookV 版本
version-proxy.php?tool=scripthookvdotnet → 返回 ScriptHookVDotNet 版本
barrage.php?action=init                  → 初始化预设弹幕
barrage.php?max=100                      → 返回弹幕列表
```

如果选「参数不跟随」或「回源跟随」：
- `$_GET` 在 PHP 中为空
- 所有请求都会走到参数校验的 400 错误分支
- CDN 缓存了这个 400 响应，后续所有请求都返回错误

#### 三种模式对比

| 模式 | 缓存时是否考虑参数 | 回源时是否携带参数 | 适用场景 |
|---|---|---|---|
| 参数不跟随（默认） | ❌ 忽略 | ❌ 丢弃 | 纯静态站，所有页面内容与参数无关 |
| 回源跟随 | ❌ 忽略 | ✅ 携带 | 静态站但有少量动态接口（缓存键仍不含参数，会串内容） |
| **全程跟随** | ✅ 考虑 | ✅ 携带 | **PHP API 站、动态内容站（本项目）** |

---

### 不缓存规则

为所有 PHP API 设置不缓存规则，让请求直接穿透到源站：

| 配置项 | 值 |
|---|---|
| 类型 | 路径 |
| 规则 | `/api/*.php` |
| 状态码 | `200-499` |

#### 为什么状态码要填 `200-499`

默认只匹配 `200 ~ 399`（不含 301、302）。但 SORA-HUB 的 API 会返回：

| 状态码 | 场景 | 是否需要不缓存 |
|---|---|---|
| 200 | 正常响应 | ✅ 避免缓存过期数据 |
| 400 | 参数校验失败 | ✅ **必须**，否则错误响应被缓存后所有请求都报错 |
| 403 | 禁止访问 | ✅ 避免封禁状态被缓存 |
| 404 | 文件不存在 | ✅ 避免临时 404 被缓存 |
| 500 / 502 / 503 | 服务器错误 | ✅ 避免故障状态被缓存 |

如果不填 400，CDN 会缓存参数校验失败的 400 响应，导致正确的请求也返回 400。

---

### 缓存规则

为静态资源配置长期缓存：

| 路径/后缀 | 缓存时间 | 说明 |
|---|---|---|
| `*.css`、`*.js` | 7 天 | 前端资源带版本号或哈希，长期缓存安全 |
| `*.png`、`*.jpg`、`*.svg`、`*.webp` | 30 天 | 图片几乎不变 |
| `*.woff2`、`*.woff`、`*.ttf` | 30 天 | 字体文件不变 |
| `/assets/*`、`/img/*`、`/fonts/*` | 30 天 | 按目录兜底 |

> **注意**：`index.html`、`style.css`、`version-detector.js` 是经常修改的文件。如果你的 HTML 文件名不含哈希，建议将这些文件的缓存时间设短（10 分钟或不缓存），或者在部署后手动刷新 CDN 缓存。

---

### 浏览器缓存

浏览器缓存控制的是**用户浏览器 ↔ CDN 节点**这一段的缓存行为。

```
用户浏览器 ←(浏览器缓存)→ CDN 节点 ←(CDN 缓存)→ 源站
```

#### 与 PHP Cache-Control 的关系

如果你的 PHP 代码设置了 `Cache-Control: no-store`，但又拍云开启了「浏览器缓存 4 小时」，CDN 会在返回给浏览器的响应头里**覆盖** PHP 的设置：

```
PHP 原始头:     Cache-Control: no-store, no-cache
CDN 覆盖后:     Cache-Control: max-age=14400
```

结果：PHP 数据每 2 小时更新一次，但用户浏览器 4 小时才去验证，中间 2 小时会显示过期数据。

#### 推荐配置

- **如果支持按路径配置**：仅对 `*.css`、`*.js`、`*.png` 等静态资源开启浏览器缓存（4 小时），API 路径不设置
- **如果不支持按路径配置**：**不要开启**，靠 PHP 的 `Cache-Control: no-store` 和不缓存规则即可

---

## 安全加固

### TLS 配置

| 配置项 | 推荐值 | 原因 |
|---|---|---|
| TLS 1.3 | 开启 | 更快的握手、更强的加密，所有现代浏览器均支持 |
| 最低 TLS 版本 | TLSv1.2 | TLSv1.0/1.1 已被弃用，存在安全漏洞 |
| HTTP/2 | 开启 | 多路复用，减少连接数 |
| HTTP/3 (QUIC) | 开启 | 基于 UDP 的新一代协议，连接建立更快 |

### CC 防护

保护 PHP API 免受高频恶意请求：

| 配置项 | 值 | 说明 |
|---|---|---|
| URI 匹配规则 | `/api/*.php` | 只保护动态接口，不影响静态资源 |
| 访问频率阈值 | 200 次/分钟（平台最低值） | 正常用户不可能每秒 3 次以上请求 API |
| 防护模式 | 智能防护 | 达到阈值才触发 JS Challenge，正常访问零影响 |

#### 智能防护 vs 强制防护

| 模式 | 行为 | 适用场景 |
|---|---|---|
| 智能防护 | 达到阈值后触发 JS Challenge | ✅ 本项目使用 |
| 强制防护 | 每个请求都触发 JS Challenge | 遭受攻击时临时开启 |

> **重要**：不要对 API 路径使用强制防护。前端 JavaScript 的 `fetch()` 请求无法通过 JS Challenge，会导致版本检测和弹幕功能全部失效。

### IP 访问限制

| 配置项 | 值 | 说明 |
|---|---|---|
| 资源路径 | `/api/*` | 只限制 API，不限制静态资源 |
| 限制策略 | 单 IP 每分钟 ≤ 60 次 | 兜底防护，比 CC 防护更宽松 |

### WAF 保护

| 配置 | 推荐 | 原因 |
|---|---|---|
| WAF | 开启 | 防御 SQL 注入、XSS 等 Web 攻击 |

### HTTP 请求体大小限制

| 配置 | 推荐值 | 原因 |
|---|---|---|
| 请求体大小 | 1 MB | 弹幕接口最大 50 字，不需要 512MB |

### CORS 跨域共享

如果你的前端 JS 需要从其他域名调用 API，需要开启 CORS。SORA-HUB 的 PHP 代码已通过 `header('Access-Control-Allow-Origin: *')` 处理了 CORS，但建议在 CDN 层也确认不拦截 CORS 头。

## 性能优化

| 配置项 | 推荐 | 说明 |
|---|---|---|
| 智能压缩（Gzip + Brotli） | 开启 | 减少传输体积，Gzip 对 HTML/CSS/JS 压缩率 60-80% |
| 页面压缩 | 开启 | 自动去除 HTML/CSS/JS 中的空白和注释 |
| 重定向跟随 | 开启 | Nginx 的 HTTP→HTTPS 301 跳转由 CDN 内部处理，用户少一次往返 |
| HTTP 302 调度 | 关闭 | SORA-HUB 无此需求 |
| 设备适配 | 关闭 | 已通过 Tailwind CSS 的响应式类实现 |
| 离线模式 | 关闭 | 源站宕机时不应返回过期的 API 数据 |

## 踩坑记录

### 问题 1：版本检测始终返回 400

**现象**：浏览器控制台报 `Failed to load resource: the server responded with a status of 400`，即使 PHP 代码中的 `$_GET` 读取逻辑完全正确。

**排查过程**：

1. 重启 PHP-FPM → 无效
2. 禁用 OPcache → 无效
3. 清除 Nginx 代理缓存 → 无效
4. 在服务器上创建测试文件 `test.php`，输出 `$_GET` → 发现 `"get":[]`（空数组）
5. 用 `dig` 命令发现域名解析到 CDN IP（`nm.aicdn.com`），说明服务器本地 curl 也经过 CDN

**根因**：又拍云默认「参数不跟随」，CDN 在回源时丢弃了 URL 中 `?` 之后的所有参数，PHP 的 `$_GET` 始终为空。

**修复**：将「参数跟随」改为「全程跟随」。

### 问题 2：CDN 缓存了 400 错误响应

**现象**：即使参数跟随已修改、URL 刷新已执行，仍然返回旧的 400 响应。

**根因**：CDN 缓存了之前的 400 错误响应。不缓存规则的状态码范围只覆盖了 `200-399`，400 响应没有被排除在缓存之外。

**修复**：将不缓存规则的状态码改为 `200-499`，覆盖所有可能的响应状态码。

### 问题 3：更新代码后不生效

**现象**：通过 FTP/面板上传了新的 PHP 文件，但访问时仍是旧代码的行为。

**排查清单**（按优先级）：

1. ~~OPcache~~ — 重启 PHP-FPM 应清除，但 `opcache.revalidate_freq` 设为 0 更保险
2. ~~Nginx 缓存~~ — 检查 `proxy_cache_dir` 和 `fastcgi_cache` 配置
3. **CDN 缓存** — 最常见的原因，执行 URL 刷新
4. **CDN 参数跟随** — 如果 `$_GET` 为空，检查此配置

## 验证清单

配置完成后，用以下命令逐项验证：

```bash
# 1. 确认版本检测正常（应返回 ok:true）
curl -s "https://你的域名/api/version-proxy.php?tool=scripthookv" -k

# 2. 确认弹幕 API 正常（应返回 JSON 数组）
curl -s "https://你的域名/api/barrage.php?max=3" -k

# 3. 确认 CDN 没有缓存 API 响应（多次请求 version 字段应一致）
curl -s "https://你的域名/api/version-proxy.php?tool=scripthookv" -k

# 4. 确认静态资源被缓存（应有 CDN 缓存头）
curl -sI "https://你的域名/style.css" -k | grep -i "cache\|hit\|age"
```

全部通过后，CDN 配置即完成。
